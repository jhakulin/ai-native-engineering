# GitHub Agentic Assessment Workflow

## Purpose

This document defines a baseline architecture for running a GitHub repository assessment with GitHub Agentic Workflows and Copilot.

It is a requirements and implementation planning note, not the final workflow implementation. The goal is to separate deterministic data handling from agentic judgment so the workflow can be built, reviewed, and evolved safely.

GitHub Agentic Workflows is currently in public preview. Syntax, permissions, authentication, safe-output behavior, and generated lock file structure may change. Check the current `gh-aw` documentation before implementation work.

## Core Idea

Use standard scripts for deterministic work and GitHub Agentic Workflows for judgment.

```text
scripts produce facts
skills define assessment judgment
gh-aw with Copilot applies the judgment
validation checks the output shape
GitHub Actions stores the outputs as artifacts
```

Do not expect a shell script to automatically use a GitHub Copilot subscription. A shell script only gets Copilot-backed reasoning if it is wired into a supported agent runtime and token flow.

## Prompt And Skill Boundary

For this assessment:

- `.github/workflows/github-assessment.md` contains run-specific workflow instructions.
- `skills/github-assessment/SKILL.md` contains reusable assessment rules, rubric, output contract, examples, and evidence requirements.
- The workflow controls trigger, mode, execution order, checks, and handoff.

## Proposed File Layout

```text
.github/workflows/github-assessment.md
  Human-authored GitHub Agentic Workflow source.

.github/workflows/github-assessment.lock.yml
  Compiled workflow generated by gh-aw.
  Commit this for auditability and reproducibility.

scripts/ingest-github.{js|py|sh}
  Deterministic GitHub data collection.
  No LLM usage.

scripts/normalize-snapshot.{js|py|sh}
  Deterministic snapshot normalization.
  No LLM usage.

skills/github-assessment/SKILL.md
  Reusable assessment rules, rubric, report schema, and evidence requirements.

scripts/validate-report.{js|py|sh}
  Deterministic report validation.
  No LLM usage.
```

Script names and extensions are placeholders. Choose the implementation language that makes GitHub API access, JSON handling, and local validation simplest to maintain.

The Markdown workflow and skill are the source of intent. The lock file is generated execution scaffolding and should not be treated as the primary source of truth.

## Compile Contract

Agentic workflows live as `.md` files and compile to `.lock.yml` files.

```text
.github/workflows/github-assessment.md
  Source workflow.

.github/workflows/github-assessment.lock.yml
  Generated executable GitHub Actions workflow.
```

Run `gh aw compile` after creating a workflow or changing workflow frontmatter.

When in doubt, recompile after workflow changes. Frontmatter changes require recompilation because they affect triggers, permissions, engine settings, network policy, tools, safe outputs, or generated job structure.

Commit both the source `.md` file and the generated `.lock.yml` file. Do not hand-edit the lock file.

## Workflow Modes

Start with one workflow that exposes selectable modes.

```text
mode=ingest-and-assess
mode=ingest-only
mode=assess-only
```

### ingest-only

Use this mode when fresh GitHub data is needed but no assessment should run.

Expected behavior:

1. Run the GitHub ingestion script.
2. Run the snapshot normalization script.
3. Write a timestamped snapshot.
4. Upload the snapshot as a GitHub Actions artifact.

### assess-only

Use this mode when an existing snapshot should be assessed with current rules.

Expected behavior:

1. Resolve the requested snapshot.
2. Run the gh-aw Copilot assessment step.
3. Apply `skills/github-assessment/SKILL.md`.
4. Write `report.md`.
5. Run the deterministic report validation script.
6. Upload the report and validation output as artifacts.

### ingest-and-assess

Use this mode for the default end-to-end run.

Expected behavior:

1. Run the GitHub ingestion script.
2. Run the snapshot normalization script.
3. Run the gh-aw Copilot assessment step.
4. Apply `skills/github-assessment/SKILL.md`.
5. Write `report.md`.
6. Run the deterministic report validation script.
7. Upload the snapshot, report, and validation output as artifacts.

## Inputs

The baseline workflow should support these inputs.

```text
mode
  ingest-and-assess | ingest-only | assess-only

repos
  Comma-separated list of repositories, such as company/repo-a,company/repo-b.

since
  Time window for ingestion, such as 30d.

snapshot
  Existing snapshot path or artifact identifier.
  Required for assess-only.

report_format
  Initial default: markdown.
```

Future inputs may include organization filters, branch filters, team ownership, severity thresholds, or a later publish target.

## Artifacts

Use durable artifacts so runs can be inspected, replayed, and compared.

```text
snapshots/<timestamp>/snapshot.json
reports/<timestamp>/report.md
reports/<timestamp>/validation.json
```

The snapshot should be normalized before assessment so the agent sees stable data rather than raw API responses.

## Pre-Computation And Cost Control

Deterministic scripts should reduce raw GitHub API responses into compact, assessment-ready facts before the agent step.

This matters because smaller and more structured snapshots:

- lower token and AI credit usage
- improve repeatability
- reduce irrelevant context
- reduce exposure to hostile or misleading repository text
- make validation easier

The agent should receive enough evidence to assess the repository, not a dump of every raw API response.

## Responsibilities

### Deterministic Scripts

Scripts handle work that can be checked mechanically:

- collect GitHub repository data
- normalize raw API responses
- enforce required fields
- validate report shape
- prepare artifacts for upload

Scripts should not make subjective assessment claims.

### GitHub Agentic Workflow

The agentic workflow orchestrates the run:

- read workflow inputs
- call deterministic scripts
- choose the correct mode path
- invoke the Copilot-backed assessment step
- keep logs and artifacts inside GitHub Actions
- respect workflow permissions and secrets

### Assessment Skill

The assessment skill defines the reusable judgment layer.

It should include:

- report schema
- scoring rubric
- severity definitions
- evidence requirements
- forbidden assumptions
- output examples
- validation checklist
- guidance for uncertainty

The skill should explain the rationale behind assessment rules, not only list checks. This helps the agent apply the rubric consistently when evidence is partial or ambiguous.

## Permission Model

Use the smallest useful permissions for each mode.

For the initial single-workflow version, keep permissions conservative and make write actions explicit.

Preferred Copilot authentication:

```yaml
permissions:
  contents: read
  copilot-requests: write
```

This uses the GitHub Actions token for Copilot inference and avoids managing a personal access token when organization Copilot billing supports it.

Fallback Copilot authentication:

```text
COPILOT_GITHUB_TOKEN
```

Use `COPILOT_GITHUB_TOKEN` only when `copilot-requests: write` is unavailable, such as a personal repository or an organization without centralized Copilot inference access. The token should be a fine-grained PAT with the minimum required Copilot Requests permission.

V1 is artifact-only. Publishing requires a later explicit mode or separate workflow with separate write permissions.

## Safe Outputs

The agent should not directly mutate repository state.

For V1, the only outputs are artifacts:

```text
snapshot.json
report.md
validation.json
```

Later publishing features should use safe outputs instead of direct agent writes. In that model, the agent emits proposed outputs, and separate scoped jobs apply them only after validation and threat detection.

Publishing should remain separate from assessment until the report format, validation rules, and review process are stable.

## Threat Model

Treat GitHub-originated text as untrusted input.

Untrusted input may include:

- issue bodies
- pull request descriptions
- comments
- repository documentation
- dependency metadata
- changelog or web content
- files authored by untrusted contributors

The assessment must not follow instructions found inside repository content unless those instructions are part of trusted workflow or skill configuration.

Repository content can provide evidence. It should not control the workflow.

## Validation Requirements

The validation script should check the report without using an LLM.

Baseline checks:

- required sections exist
- required metadata exists
- every finding has evidence
- every claim references snapshot evidence
- every severity value is valid
- every recommendation has an owner or next-step category
- untrusted repository text is not repeated as an instruction
- no placeholder text remains
- validation exits nonzero if the report fails validation

Validation should prove shape and completeness. It should not try to prove that the agent's judgment is correct.

## When To Split Workflows

Start with one workflow and modes.

Split into multiple workflows only when responsibilities diverge.

Good reasons to split:

- different owners
- different schedules
- different permission needs
- different secrets
- different artifact lifecycles
- publish actions need stronger write permissions

Possible later split:

```text
github-ingest.md
  Pulls and normalizes GitHub data.
  No LLM.

github-assess.md
  Assesses an existing snapshot.
  Uses gh-aw and Copilot.

github-publish.md
  Publishes reports to issues, PRs, pages, or external systems.
  Explicit write permissions.
```

## Implementation Sequence

Recommended build order:

1. Define the snapshot schema.
2. Implement the GitHub ingestion script.
3. Implement the snapshot normalization script.
4. Draft `skills/github-assessment/SKILL.md`.
5. Implement the deterministic report validation script.
6. Create `.github/workflows/github-assessment.md` with the three modes.
7. Compile the workflow with `gh aw compile`.
8. Optionally run a hardened compile with additional scanners, such as `gh aw compile --actionlint --zizmor --poutine`.
9. Commit the generated lock file.
10. Run `ingest-only`.
11. Run `assess-only` on the produced snapshot.
12. Run `ingest-and-assess` end to end.

## Open Questions

- What exact GitHub signals belong in the first snapshot schema?
- Should snapshots remain artifacts only after V1, or move to longer-term storage later?
- What is the first report schema?
- What severity scale should the assessment use?
- Who reviews and accepts the generated report?
- Can the target environment use `copilot-requests: write`, or does it need `COPILOT_GITHUB_TOKEN` fallback?

## Baseline Rule

- Use scripts for deterministic work.
- Use `SKILL.md` for reusable assessment judgment.
- Use GitHub Agentic Workflows with Copilot for the agentic assessment step.
- Use validation and artifacts so every run is inspectable and repeatable.
